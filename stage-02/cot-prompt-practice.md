# Stage 2 - CoT Prompt Practice

## 1. CoT 是什么

CoT 是 Chain of Thought，可以理解为让模型在复杂任务中先检查关键条件，再给出结论。

在实际使用中，不建议要求模型输出完整思考过程。更推荐要求模型：

- 检查关键条件
- 应用规则
- 输出结论
- 给出简短理由

## 2. CoT 适合什么时候用

CoT 适合：

- 分类规则有冲突
- 任务需要多步判断
- 需要比较几个选项
- 需要排查错误原因
- 直接回答容易漏条件

CoT 不适合：

- 任务很简单
- 只需要固定格式输出
- 只是翻译、改写、提取字段
- 输出必须非常短

## 3. 我的 CoT Prompt

```text
角色：
你是一个学习记录分类助手。

任务：
请根据分类规则和检查步骤，把待分类输入归入一个最合适的类别。

类别列表：
只能从以下类别中选择一个：
- Git
- Security
- Other

分类规则：
1. 如果内容涉及 git status、git add、git commit、.gitignore，选择 Git。
2. 如果内容涉及 `.env`、`.env.example`、API key、密码、隐私文件、密钥管理，选择 Security。
3. 如果同时涉及 Git 忽略规则和密钥安全，优先选择 Security。
4. 如果无法判断，选择 Other。

检查步骤：
1. 检查是否涉及安全内容。
2. 检查是否涉及 Git 内容。
3. 如果同时涉及 Git 和安全，应用安全优先规则。
4. 不要输出完整检查过程，只输出最终类别和简短理由。

待分类输入：
今天我学习了 .gitignore 规则和 .env.example 模板。

输出格式：
类别：
简短理由：

限制：
不要读取、输出、修改 `.env`、API key、密码文件或隐私文件。

. 预期输出
类别：Security
简短理由：同时涉及 Git 忽略规则和密钥安全模板，按安全优先规则分类。

5. 我的理解
CoT 的重点不是让模型输出完整思考过程，而是让模型先检查关键条件，再给出更稳的结论。
在分类任务中，如果输入同时符合多个类别，就可以用检查步骤让模型先判断是否有冲突，再应用优先级规则。
用户通常只需要最终类别和简短理由，不需要完整推理过程。

# Stage 1 - Prompt Structure Practice

## 1. 弱 Prompt 的问题

原始 prompt：

```text
帮我看看这个项目怎么优化
这个 prompt 的问题：
没有指定角色，不知道模型应该从什么角度回答。
没有提供背景，不知道项目当前处于什么阶段。
没有指定读取范围，容易让 Agent 读取过多无关文件。
没有安全边界，没有说明不能读取 .env、API key、密码文件。
没有输出格式，回答容易变得松散。
没有验收标准，模型不知道什么样的输出才算合格。
没有优先级，不知道应该优先优化计划、代码、结构、性能还是学习路线。
2. 完整结构化 Prompt
角色：
你是一个有经验的 IT 项目经理和代码审查助手。

任务：
请审查指定项目的任务计划书和已完成代码，找出不合理、不清楚或可能影响交付的地方，并给出修改建议。

背景：
我刚起草完任务计划书，并完成了第一版代码。现在需要确认计划和代码是否匹配，是否有明显遗漏。

范围：
只读取以下两个文件：
1. E:\work\project-plan.md
2. E:\work\main.py

不要读取其他文件。

安全限制：
不要读取 `.env`、API key、密码文件、隐私文件。
不要修改任何文件，只做审查和建议。
如果需要更多文件，请先说明原因，等待确认。

输出格式：
请用中文输出，按以下结构：
1. 总体判断
2. 发现的问题列表
3. 每个问题的原文依据
4. 为什么这是问题
5. 修改建议
6. 优先级：高 / 中 / 低

验收标准：
每条建议必须能对应到具体原文或具体代码位置。
不要泛泛而谈。
不要直接改文件。
3. 省 Token 版本 Prompt
角色：你是有项目经验的 IT 项目经理和代码审查助手。

任务：请审查 `123.md` 和 `123.py` 两个文件的不合理之处。

范围：只读取 `E:\work\123.md` 和 `E:\work\123.py`，不要读取其他文件。

安全限制：不要读取 `.env`、API key、密码文件、隐私文件；不要修改任何文件。

输出格式：用 Markdown 表格输出，列为：问题、优先级、依据、原因、修改建议。

验收标准：每条问题都必须对应到具体文件内容或代码位置，不要泛泛而谈。
4. 我的理解
Prompt 不只是“提问题”，而是给模型的一份任务说明。
一个好的结构化 prompt 至少应该包含：
角色
任务
背景
范围
安全限制
输出格式
验收标准
如果任务涉及 CLI Agent 或本地文件，必须明确：
读取哪些文件
不读取哪些文件
是否允许修改文件
修改前是否需要先说明计划
省 token 的 prompt 不是删掉关键限制，而是保留最重要的信息，去掉不必要的解释。
